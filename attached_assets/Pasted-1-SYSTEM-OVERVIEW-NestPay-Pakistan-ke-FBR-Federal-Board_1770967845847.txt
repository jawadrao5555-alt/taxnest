1. SYSTEM OVERVIEW
NestPay Pakistan ke FBR (Federal Board of Revenue) ke PRAL DI API v1.12 ke mutabiq digital invoices banata aur submit karta hai. Yeh SRO 709(I)/2025 compliant hai.

Dual Mode Operation:

Sandbox Mode → Testing ke liye (FBR test server pe submit hota hai)
Production Mode → Real FBR submission (final, immutable, locked)
Multi-Company Support:

Har company ke apne FBR tokens hain (Sandbox + Production alag)
Data isolation: Ek company doosri ka data nahi dekh sakti
Tax logic GLOBAL hai — ek formula sab companies ke liye
2. INVOICE LIFECYCLE (Step by Step)
User Invoice Banata Hai (Draft)
        ↓
  [DRAFT STATUS] — Edit/Delete Allowed
        ↓
  User "Send to Sandbox" Click Karta Hai
        ↓
  [SANDBOX STATUS] — FBR Test Server pe Submit
        ↓
  User "Send to Production" Click Karta Hai
        ↓

        ↓
  ❌ LOCKED — Edit/Delete Impossible

Method: POST
Content-Type: application/json

5. ITEM-LEVEL FIELDS EXPLAINED
Field	Type	Description
uoM	string	Unit of Measure — "U" (Units), "KG", "Liter", "M", "SQM"
rate	string	Tax rate as percentage — "18%", "5%", "0%", "Exempt"
hsCode	string	HS Code in XXXX.XXXX format — e.g., "2202.1010"
discount	number	Discount amount (default 0.00)
extraTax	number	Extra tax (0.00 for standard, "" for reduced/exempt)
quantity	number	Quantity (4 decimal places)
saleType	string	Sale category (see mapping below)
fedPayable	number	Federal Excise Duty (default 0.00)
furtherTax	number	Further tax amount (default 0.00)
totalValues	number	valueSalesExcludingST + salesTaxApplicable
productDescription	string	Item description
salesTaxApplicable	number	Calculated tax amount
valueSalesExcludingST	number	Amount before tax
salesTaxWithheldAtSource	number	WHT at source (default 0.00)
fixedNotifiedValueOrRetailPrice	number	MRP/Retail price (3rd Schedule tax base)
Conditional SRO Fields (only when applicable):

Field	When Required
sroScheduleNo	3rd Schedule (rate < 18%), Exempt, Reduced Rate
sroItemSerialNo	Same as above — serial number in the schedule
scenarioId	ONLY in Sandbox mode (SN001-SN028)

7. TAX CALCULATION ENGINE (4 Formulas)
┌─────────────────────────────────────────────────────────────┐
│ STANDARD RATE (18%)                                          │
│ valueSalesExcludingST = quantity × unitPrice                 │
│ salesTax = valueSalesExcludingST × rate%                     │
│ Example: 10 × 1000 = 10000 → Tax = 10000 × 18% = 1800      │
├─────────────────────────────────────────────────────────────┤
│ 3RD SCHEDULE GOODS (5%, 10%, 12%, 18%)                       │
│ Tax Base = fixedNotifiedValueOrRetailPrice (MRP)             │
│ salesTax = retailPrice × rate%                               │
│ Example: MRP = 5000 → Tax = 5000 × 5% = 250                 │
│ SRO RULE: rate < 18 → sroScheduleNo = "3rd Schedule goods"  │
│           rate = 18 → NO SRO fields sent                     │
├─────────────────────────────────────────────────────────────┤
│ REDUCED RATE (8th Schedule)                                  │
│ salesTax = valueSalesExcludingST × reducedRate%              │
│ extraTax = "" (empty string, NOT 0)                          │
│ sroScheduleNo = "EIGHTH SCHEDULE Table 1"                    │
├─────────────────────────────────────────────────────────────┤
│ EXEMPT GOODS (6th Schedule)                                  │
│ salesTax = 0                                                 │
│ rate = "Exempt" (NOT "0%")                                   │
│ extraTax = "" (empty string, NOT 0)                          │
│ sroScheduleNo = "6th Schd Table I"                           │
└─────────────────────────────────────────────────────────────┘

10. IMPORTANT BUSINESS RULES
Rule	Detail
CNIC/NTN Format	Remove ALL dashes/spaces — CNIC = 13 digits, NTN = 7 or 9 digits
Buyer Unregistered	buyerNTNCNIC = "" (empty string, field MUST exist)
Credit Notes	NEVER sent to FBR (internal-only). Debit Notes are used for FBR returns
Production Lock	Once submitted to Production, invoice is PERMANENT — no edit, no delete
HS Code Format	XXXX.XXXX format (e.g., "2202.1010") — add dot between 4th and 5th digit
UoM for Beverages	Chapter 22 HS Codes → UoM = "Liter" (full word)
Sanitization	Remove newlines, tabs, extra spaces from all text fields
items FIRST	In the JSON payload, "items" array must be the FIRST key
11. FBR ERROR CODES
Code	Meaning
0001	Seller not registered for sales tax
0002	Invalid Buyer NTN/CNIC format
0003	Invalid invoice type
0005	Date format wrong (need YYYY-MM-DD)
0009	Buyer registration number missing
0010	Buyer name empty
0013	Sale type invalid
0019	HS Code missing
0020	Rate field empty
0021	Value of Sales / Quantity empty
0401	Seller unauthorized (invalid token)
0402	Buyer unauthorized

13. COMPLETE SUBMISSION FLOW (Code Level)
Step 1: User clicks "Submit to Production"
Step 2: Frontend calls → POST /api/invoices/:id/submit-to-fbr
Step 3: Backend loads invoice + items + company data
Step 4: Checks: Credit Note? → BLOCK | Already Production? → BLOCK
Step 5: transformToFbrJson() → Builds exact FBR payload
Step 6: validateFbrJson() → Checks all required fields
Step 7: axios.post(FBR_URL, payload, { Authorization: Bearer token })
Step 8: FBR Response → statusCode "00"? 
         ↳ YES → Update invoice status = "production", save FBR invoice number
         ↳ NO  → Return error message to user, invoice stays draft
Step 9: If FBR down → Queue for auto-retry
